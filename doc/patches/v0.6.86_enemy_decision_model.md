# v0.6.86 proposal — source-first enemy decision model scaffold

This package is intentionally additive. It adds a new C# class:

```text
scripts/tools/simulation/LadyBugEnemyDecisionModel.cs
```

It does **not** modify `LadyBugSimulationState.cs` yet, and it does not remove any current reference-sync bridge.

## Why additive first?

The current checkpoint keeps the standard JSONL comparison passing while `preferred[]`, `rejectedMask`, fallback helper, enemy direction, chase timers, and round-robin are still partly reference-synced. Replacing that pipeline before validating the individual Z80 blocks would reintroduce the old problem: fixing mismatches by fitting logs.

This class is meant to be used as a block-level validation harness first.

0x3911..0x3955  ValidateLogicalMazeDirection()
0x4130..0x4188  ValidateLocalDoorBlock()

## Important caveats

### 1. `EnemyIsAtDecisionCenter()`

The function currently implements the pixel-alignment predicate:

```text
(x & 0x0F) == 0x08
(y & 0x0F) == 0x06
```

The later `0x428D..0x42B8` code path calls helper routines that still need focused validation. The method is therefore intentionally documented as the alignment part of `0x427E`, not as the full carry behavior of the original routine.

### 2. `GetTileUnderPlayerProbe` is delegated

`0x4130` and `0x4189` call `GetTileUnderPlayerProbe` at `0x3C0A`. This package does not guess that address conversion. Instead, it accepts:

```csharp
Func<int, int, int> readTileAtProbe
```

The next step is to plug this into the existing trace/maze representation, or to transcribe `0x3C0A` separately.

### 3. Logical-maze table

`0x3911` reads a packed table at arcade address `0x0DA2`, not the dynamic BFS RAM at `0x6200`. The class therefore expects:

```csharp
IReadOnlyList<int> packedMazeTable0DA2
```

The next step is to extract that table into a stable fixture so `ValidateLogicalMazeDirection()` can be tested directly.

### 4. `0x43BA` loop is not fully integrated

`ApplyEnemyTempMovementStep()` implements the atomic part:

```text
call 0x4224
inc 0x61C2
```

The stack-dependent loop at `0x43C5..0x43CB` still needs validation before being used to drive full enemy updates.

## Recommended next validation

1. Extract the packed `0x0DA2` table into a small fixture.
2. Add a tiny block-level check for `ValidateLogicalMazeDirection()` using exact-PC `0x3911` hits.
3. Add a tile-probe adapter for the current trace/maze representation.
4. Validate `ValidateLocalDoorBlock()` against exact-PC `4130_LOCAL_DOOR_CHECK_ENTRY` and `4187_LOCAL_DOOR_REJECT` events.
5. Only after that, call `TryPreferredDirection()` from the adapter in shadow mode.

## Installation

Copy the package contents into the repository root:

```text
EnemyTraceSimulator/
└─ scripts/tools/simulation/LadyBugEnemyDecisionModel.cs
```

Godot's .NET project includes `.cs` files automatically, so no `.csproj` edit is needed.
