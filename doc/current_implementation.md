## Purpose of this document

The README gives a concise project overview. This file keeps the technical detail.

## 1. Project goal

Enemy Trace Simulator is a standalone Godot/.NET diagnostic tool for validating the enemy movement logic of the Lady Bug arcade remake.

The intended final workflow is:

1. run Lady Bug in MAME;
2. load a known MAME save-state;
3. export deterministic reference data;
4. load the reference trace in Godot;
5. initialize the C# enemy simulation from the same state;
6. compare the C# simulation against MAME frame by frame;
7. replace reference-synced pieces only after their source-first model has been validated.

The project does not yet run fully independent arcade enemy AI. The current adapter remains reference-assisted by design.

## 1.1 Source-first rule for enemy decisions

New enemy decision logic must be implemented from the disassembled source first.

Primary source:

```text
LadyBug_enemy_management_extract.txt
```

MAME traces and exact-PC logs are validation tools. They should confirm whether the C# transcription matches the arcade code, but they should not be used to create case-by-case simulation rules.

Avoid adding logic such as:

```csharp
if (currentDir == 0x01 && preferred0 == 0x08) ...
```

or:

```csharp
if (referenceRejectedMask == 0) ...
```

unless the condition is directly justified by the disassembled Z80 code.

The correct development loop is:

1. identify the Z80 block and RAM writes in `LadyBug_enemy_management_extract.txt`;
2. transcribe that block into a source-named C# function;
3. validate the function against standard JSONL traces and exact-PC logs;
4. only then expand coverage to new traces;
5. avoid filtering or hiding mismatches unless the source code explains why that state is not comparable.

## 2. Current validation strategy

There are five complementary diagnostic streams.

### 2.1 Standard JSONL trace

Generated by:

```text
tools/mame/lua/ladybug_sequence_trace.lua
```

Used for:

- normal trace loading;
- visual playback;
- comparison against C# simulation frames;
- safe polling-based `preferredChangeEvents`;
- adapter-level preferred[] shadow compare;
- adapter-level `rejectedMask` shadow compare;
- adapter-level fallback helper / `0x61C2` shadow compare;
- adapter-level enemy direction shadow compare;
- source-first transition diagnostics over frame-level state;
- source-first release diagnostics over the first activation transition.

This is the main validation pipeline.

### 2.2 Exact-PC preferred[] diagnostic

Generated by:

```text
tools/mame/lua/ladybug_preferred_pc_trace.lua
```

Used for:

- reverse-engineering exact writes to `EnemyWork.preferred[]`;
- identifying the CPU instruction that wrote a byte;
- capturing `R`, `A`, `HL`, `IY`, chase timers, and existing preferred[] values at the breakpoint;
- validating the standalone `LadyBugMonsterPreferenceSystem`;
- shadow-replaying the preferred[] write stream before adapter integration.

Raw output:

```text
tools/mame/lua/error.log
```

Automatic report:

```text
traces/mame/<outputPrefix>_preferred_pc_analysis.txt
```

This is not the normal comparison format. It is an exact-PC reverse-engineering aid.

### 2.3 Exact-PC EnemyWork decision diagnostic

Generated by:

```text
tools/mame/lua/ladybug_enemywork_pc_trace.lua
```

Used for:

- reverse-engineering `EnemyWork.rejectedMask`;
- identifying how the fallback helper at `0x61C2` behaves;
- classifying `Enemy_UpdateOne` decision cycles;
- detecting local-door rejection, generic fallback, and forced reversal.

Raw output:

```text
tools/mame/lua/error.log
```

Automatic report:

```text
traces/mame/<outputPrefix>_enemywork_pc_analysis.txt
```

This is also not a JSONL trace. It should be read as an exact-PC report.

### 2.4 Source-first decision diagnostics

Implemented by:

```text
scripts/tools/simulation/LadyBugEnemyDecisionModel.cs
scripts/tools/simulation/LadyBugStaticMazeRomTable.cs
scripts/tools/simulation/LadyBugEnemyDecisionTraceDiagnosticAdapter.cs
```

Used for:

- running reconstructed decision functions over the loaded MAME trace;
- validating `0x3911`, `0x42E6`, and `0x4241` behavior progressively;
- recording where modeled scratch state differs from MAME scratch state;
- keeping the comparison non-invasive while the decision model is incomplete.

Important limitation:

```text
The diagnostic currently uses a permissive local-door tile reader.
It does not yet faithfully validate 0x4130 because 0x3C0A tile lookup is still missing.
```

### 2.5 Source-first release diagnostics

Implemented by:

```text
scripts/tools/simulation/LadyBugEnemyReleaseModel.cs
```

Used for:

- identifying the first transition where an enemy slot becomes active;
- comparing the observed slot state against the source initialization shape from `0x3061`;
- detecting whether the observed slot looks like `0x3061` plus one movement/update step;
- keeping the den-exit problem visible instead of treating it as a normal plain-step mismatch.

Important limitation:

```text
This is diagnostic only. It does not yet activate an enemy slot, simulate the release
sequence, or drive the first den-exit movement.
```

## 4. Important runtime encodings

### 4.1 Player direction at 0x6198

The player direction byte used by the preferred[] rotate branch is:

```text
0x6198 = PLAYER_DIR_CURRENT
```

Runtime checks in MAME confirmed:

```text
01 = left
02 = down
04 = right
08 = up
```

Important distinction:

```text
This mapping is the player RAM direction encoding used by the 2E97 preferred[] rotate branch.
Do not invert 02 and 08 when interpreting 0x6198.
```

### 4.2 Enemy direction encoding

Enemy movement direction encoding from `LadyBug_enemy_management_extract.txt`:

```text
01 = left
02 = up
04 = right
08 = down
```

This differs from the player RAM direction encoding.

### 4.3 Enemy slot layout

The four active enemy slots are 5 bytes each starting at `0x602B`:

```text
Enemy0 = 0x602B
Enemy1 = 0x6030
Enemy2 = 0x6035
Enemy3 = 0x603A
```

Layout:

```text
+0 = direction/flags byte. High nibble stores direction; bit 1 marks active/enabled.
+1 = X pixel coordinate.
+2 = Y pixel coordinate.
+3 = sprite-related byte.
+4 = attribute-related byte.
```

### 4.4 EnemyWork addresses currently under investigation

```text
61BD = EnemyTemp_Dir
61BE = EnemyTemp_X
61BF = EnemyTemp_Y
61C1 = EnemyRejectedDirMask
61C2 = fallback step counter/helper
61C4..61C7 = EnemyWork.preferred[]
61CE..61D1 = chaseTimers[]
61D2 = chaseRoundRobin
6200..62AF = logical maze map, 11 x 16 cells
```

Important naming note:

```text
The C# trace/state model still has a legacy name fallbackMask for 0x61C2.
The exact-PC evidence shows that 0x61C2 behaves more like a fallback step counter/helper than a normal direction mask.
```

### 8.4 Decision cases now separated

Case A: preferred rejected, current direction kept.

Pattern:

```text
4315 without 4331/fallback
```

Meaning:

```text
a preferred candidate was rejected,
but the current temp direction remained acceptable,
so the enemy kept moving in the current direction.
```

Case B: preferred rejected, current direction also rejected, fallback entered.

Pattern:

```text
4315 -> 4331 -> 4241
```

Meaning:

```text
a preferred candidate was rejected,
the current temp direction was also rejected,
then the fallback finder selected another direction.
```

Case C: first activation / release transition.

Pattern:

```text
previous frame has no active enemy slot
current frame has slot raw=82, x=58, y=87, temp=08:58,87, rejected=02
```

Meaning:

```text
This is not a normal continuation of previous EnemyWork. It should be explained by
the release path around 0x05AE / 0x3061 and the first update step.
```

### 9.3 Implemented source blocks

#### `EnemyIsAtDecisionCenter()`

Source scope:

```text
0x427E pixel-center predicate
```

Implemented predicate:

```text
(x & 0x0F) == 0x08
(y & 0x0F) == 0x06
```

Note:

```text
This is currently the simple pixel-center predicate, not necessarily the complete
carry behavior of the full 0x427E helper path.
```

#### `ValidateLogicalMazeDirection()`

Source scope:

```text
0x3911..0x3955
```

Behavior:

- computes the packed logical-maze table index from X/Y;
- selects low or high nibble from the packed table;
- checks whether `allowedMask & candidateDirection` is non-zero;
- returns accepted/rejected plus diagnostic table metadata.

The packed ROM table is supplied by:

```text
scripts/tools/simulation/LadyBugStaticMazeRomTable.cs
```

#### `ValidateLocalDoorBlock()`

Source scope:

```text
0x4130..0x4188
```

Behavior:

- applies direction-specific probe offsets;
- asks a caller-supplied tile reader for the tile at the probe position;
- rejects door-local blocking tiles.

Current limitation:

```text
The source block is scaffolded, but the diagnostic tile reader is permissive.
The real 0x3C0A tile lookup still needs to be reconstructed before this block can
be trusted against MAME.
```

#### `CheckDoorForcedReversal()`

Source scope:

```text
0x4189..0x4223
```

Behavior:

- probes door-local tiles outside the normal decision-center path;
- returns true when the arcade would force reversal.

Current limitation:

```text
Same tile-reader limitation as 0x4130.
```

#### `ReverseDirection()`

Source scope:

```text
0x4347..0x4356
```

Z80 formula:

```text
A = dir
B = dir
A >>= 2
B <<= 2
dir = (A | B) & 0x0F
```

One-hot result:

```text
01 left  -> 04 right
02 up    -> 08 down
04 right -> 01 left
08 down  -> 02 up
```

#### `MoveTempOnePixel()`

Source scope:

```text
0x4224..0x4240
```

Behavior:

- moves temp X/Y by exactly one arcade pixel according to `EnemyTemp_Dir`.

#### `FindFallbackDirection()`

Source scope:

```text
0x4241..0x427D
```

Behavior:

- starts from `EnemyRejectedDirMask`;
- scans candidate directions in order `01`, `02`, `04`, `08`;
- skips already rejected bits;
- validates through `0x3911` and `0x4130`;
- selects the first accepted candidate.

Current limitation:

```text
Because 0x4130 is still using a permissive tile reader in diagnostics, fallback
validation is not yet complete for door-local cases.
```

#### `TryPreferredDirection()`

Source scope:

```text
0x42E6..0x4337
```

Behavior:

1. test preferred direction through logical maze and local-door validation;
2. if preferred is accepted, assign `EnemyTemp_Dir = preferred`;
3. if preferred is rejected, write `EnemyRejectedDirMask |= preferred`;
4. test current temp direction against the same logical/local constraints;
5. if current is accepted, keep it;
6. if current is rejected too, write `EnemyRejectedDirMask |= current` and enter fallback.

#### `ApplyEnemyTempMovementStep()`

Source scope:

```text
0x43BA..0x43CB
```

Behavior:

- calls the one-pixel temp movement;
- increments `0x61C2` fallback helper.

Note:

```text
The stack-dependent loop around 0x43C5 is not fully modeled yet.
```

#### `LoadCurrentStateToTemp()` and `CommitTempState()`

Source scopes:

```text
0x43F0..0x4405 load current enemy state into temp
0x43D4..0x43EF commit temp state back to enemy slot
```

Behavior:

- transfer the high-nibble enemy direction and X/Y coordinates into/out of scratch state.

## 10. Source-first release model

Implemented file:

```text
scripts/tools/simulation/LadyBugEnemyReleaseModel.cs
```

### 10.1 Source basis

Relevant source blocks:

```text
0x05AE  scan enemy slots and call release initializer
0x3061  initialize released enemy slot
```

Current model captures the observed source initialization shape:

```text
raw = 0x82
x   = 0x58
y   = 0x86
```

### 10.2 Current release diagnostic result

Latest validated one-enemy trace:

```text
activationTransitions=1
sourceReleaseLikeTransitions=1
sourceReleaseAfterStepTransitions=1
```

First activation context:

```text
tick=5
mameFrame=10
slot=0
active=0->1
slotRawXY=82:58,87
expected3061=82:58,86
work=08:58,87
rejected=02
matchesRaw=True
matchesX=True
matchesY3061=False
matchesYAfterStep=True
matchesWorkAfterStep=True
```

Interpretation:

```text
The observed first active enemy does not match the raw 0x3061 initialization exactly,
because the captured Y is already 0x87 rather than 0x86. It does match 0x3061 plus
one movement/update step, and EnemyWork also matches the post-step state.
```

### 10.3 What this does not prove yet

The release diagnostic does not yet prove the full runtime order. It does not know whether the frame boundary captured:

- immediately after `0x3061`;
- after a call path from `0x05AE`;
- after an alternate caller such as the known XREF around `0x4471`;
- after a fallback pass at `0x4241`;
- after the movement step and commit at `0x43BA` / `0x43D4`.

Therefore, the release diagnostic must stay separate from the rejectedMask transition model until exact-PC release logging confirms the true sequence.

## 13. Current limitations

### 13.1 Multi-enemy traces are not official yet

The current validation target is one active enemy. Multi-enemy traces are deferred until the shared `EnemyWork` scratch state can be attributed to enemy slots more explicitly.

### 13.2 preferred[] is still reference-synced in the adapter

The standalone model and adapter shadow compare are validated on the current one-enemy traces, but the simulation adapter has not yet replaced the MAME reference-sync for authoritative `preferred[]`.

### 13.3 rejectedMask and fallback helper are still authoritative reference-synced

Authoritative `rejectedMask` and the `0x61C2` fallback helper are currently synchronized from MAME. This avoids misleading mismatches in den-exit / special movement windows while the real rejection and fallback generator is still being reconstructed.

The adapter-level rejectedMask shadow model currently matches the validated trace, but the transition diagnostic still exposes the release boundary as a non-normal movement transition.

The fallback helper shadow model currently matches the validated one-enemy trace.

### 13.4 enemy direction is still authoritative reference-synced

The direction shadow model currently matches the one-enemy trace, but authoritative enemy direction remains reference-synced until the source-based decision model is validated more broadly.

### 13.5 release / den-exit is not simulated yet

The source release diagnostic confirms that the first activation looks like `0x3061` plus one movement/update step, but the simulator does not yet independently:

- choose the free enemy slot;
- initialize it from the arcade release path;
- run the first fallback/update step from source;
- commit the first active state without MAME reference sync.

### 13.6 chase state is still reference-synced

`chaseTimers[]` and `chaseRoundRobin` remain synchronized from MAME. The exact-PC diagnostic currently proves how BFS writes into preferred[0], but not the full independent chase activation logic.

### 13.7 BFS pathfinding is not yet implemented

The exact-PC shadow replay and adapter shadow compare apply or infer BFS/chase overlays from observed reference data. They do not yet compute the BFS direction independently.

### 13.8 0x4130 local-door validation is incomplete

The source-first model contains a `ValidateLocalDoorBlock()` scaffold, but the diagnostic path still uses a permissive tile reader.

To make this block authoritative, reconstruct:

```text
0x3C0A GetTileUnderPlayerProbe / tile lookup
```

using captured VRAM, arcade coordinate mapping, and the Lady Bug VRAM column-major layout.

### 13.9 Actor rendering is diagnostic

The MAME Y mirror is good enough for the current viewer, but rendering offsets are not gameplay coordinates. Player sprite orientation follows the observed `0x6198` encoding, but sprite rendering is still diagnostic and separate from gameplay simulation.

### 13.10 Generated logs are local artifacts

Generated logs and reports should not normally be committed.
