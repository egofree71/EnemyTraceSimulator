# Enemy Trace Simulator

Enemy Trace Simulator is a standalone **Godot .NET / C#** diagnostic tool used to validate enemy movement behavior for the arcade game **Lady Bug**.

The repository is separate from the main Lady Bug remake project. Its job is not to become a second implementation of Lady Bug. Its job is to act as a microscope:

```text
MAME trace = reference
Godot / C# simulation = candidate
Enemy Trace Simulator = visual and diagnostic comparison harness
```

For detailed implementation notes, source-address mappings, reverse-engineering status, and workflow details, see:

```text
doc/current_implementation.md
```

## Current status

Current checkpoint: **v0.9.15b**

Latest stable milestone:

```text
Seeded C# random mode and settings UI for autonomous behavior observation
```

v0.9.15b keeps the v0.9.14c source-path single-enemy replay as the default exact-validation candidate, and adds an optional behavior-observation mode for the remaining `0x2EC7` random branch.

By default, random preferred[] tuples remain trace-synced for exact MAME comparison. When `preferredRandomMode` is set to `CSharpSeeded`, the left-side candidate uses a deterministic C# random source instead. Visual divergence from MAME is then expected, but the mode is useful for observing whether the simulated enemy movement behaves well without copying MAME random results.

The current validated scope is:

```text
validated scope    = one active enemy, static player, source-path movement replay
preferred[] scope  = 0x2E97 rotate modeled; visible 0x477D BFS/chase modeled; 0x2EC7 random either trace-synced or C# seeded
out-of-scope now   = autonomous release, exact Z80 R-register reproduction, moving player, true multi-enemy replay
future scope       = moving player, pivoting-door interaction, multi-enemy logic
```

The key progression is:

```text
Before v0.9.9:
  the simulation board advanced the enemy using the direction observed in the MAME trace.

Since v0.9.9b:
  the simulation board computes the one-enemy step with the reconstructed source path:
    0x427E decision gate
    -> 0x42E6 preferred / 0x4325 current / 0x4241 fallback
    -> 0x43BA one-pixel movement

Since v0.9.10b:
  this source-path replay is the default candidate exposed by the UI and adapter wrapper.

Since v0.9.12:
  preferred[] tuples produced by the deterministic 0x2E97 rotate branch are generated by the model and used by the replay.

Since v0.9.14c:
  visible 0x477D BFS/chase overrides are generated by the source model:
    enemy position
    -> 0x45DC coordinate-to-logical-maze-index conversion
    -> 0x6200 + index
    -> low nibble used as BFS guidance direction

Since v0.9.15b:
  config/enemy_simulation_settings.json can switch the 0x2EC7 random branch between:
    TraceSynced   = exact MAME comparison
    CSharpSeeded  = deterministic C# random for autonomous behavior observation
```

Still synchronized from MAME for exact-validation mode:

```text
release timing
random 0x2EC7 preferred[] base tuples when preferredRandomMode=TraceSynced
player position
gates
VRAM context
timers
inactive slots
multi-enemy frames after the scope limit
```

In `CSharpSeeded` mode, the random base tuples are generated by the C# candidate instead. This mode is not expected to remain visually identical to MAME.

So this is not yet fully autonomous enemy AI, but the default candidate now models the visible source-path movement step, deterministic rotate preferred[] generation, visible BFS/chase preferred[] overrides, and can optionally use C# random for behavior observation.

## Simulation candidate settings

The replay behavior has its own settings file, separate from MAME trace generation:

```text
config/enemy_simulation_settings.json
```

Default exact-comparison mode:

```json
{
  "preferredRandomMode": "TraceSynced",
  "csharpRandomSeed": 1981,
  "continuePlaybackOnExpectedRandomDivergence": true
}
```

Behavior-observation mode with deterministic C# random:

```json
{
  "preferredRandomMode": "CSharpSeeded",
  "csharpRandomSeed": 1981,
  "continuePlaybackOnExpectedRandomDivergence": true
}
```

These options are editable from the gear/settings window under:

```text
Simulation candidate behavior
```

When `CSharpSeeded` is enabled, differences from the MAME trace are expected. The purpose is to watch the left-side simulation behave with its own random source, for example to check whether an enemy gets stuck between logical cells or in cul-de-sac situations.

## Compare adapter

The current main Compare action is:

```text
Run Lady Bug source-path single-enemy replay
```

This mode is implemented through:

```text
LadyBugEnemySimulationAdapter
  -> LadyBugSourcePathSingleEnemyReplayAdapter
```

For one-active-enemy transitions, it computes the enemy movement step with the reconstructed source path:

```text
0x427E decision gate
  carry set:
    preferred 0x42E6
    current   0x4325 if preferred rejected
    fallback  0x4241 if current rejected
  carry clear:
    outside-center keep / forced reversal 0x433A
then:
  0x43BA one-pixel movement step
```

For preferred[] generation, v0.9.15b uses a configurable hybrid provider:

```text
0x2E97 rotate branch            -> modeled when safe
visible 0x477D BFS/chase branch -> modeled from 0x45DC + 0x6200 when safe
0x2EC7 random branch            -> TraceSynced or CSharpSeeded, depending on settings
unsupported / ambiguous cases   -> trace-synced fallback
```

## Coordinate and data conventions

### MAME-to-Godot vertical mirror

```csharp
private const int MameYMirror = 0xDD;

private static int ToMameY(int godotArcadeY)
{
    return MameYMirror - godotArcadeY;
}
```

### Enemy direction encoding

```text
01 = left
02 = up
04 = right
08 = down
```

When comparing source enemy directions to Godot `maze.json`, vertical directions must be mirrored:

```text
source 01/left  -> Godot maze left
source 02/up    -> Godot maze down
source 04/right -> Godot maze right
source 08/down  -> Godot maze up
```

### Player direction / input encoding

```text
01 = left
02 = down
04 = right
08 = up
```

Do not mix player and enemy direction encodings.

### Enemy decision center

```text
x & 0x0F == 0x08
y & 0x0F == 0x06
```

### VRAM layout

Lady Bug background VRAM is column-major and bottom-to-top:

```text
D080-D09F = first column, bottom to top
D0A0-D0BF = second column, bottom to top
...
D3E0-D3FF = last column, bottom to top
```

### BFS/chase guidance map

v0.9.14b/v0.9.14c validated the source-first BFS guidance lookup used before `0x477D`:

```text
0x45DC:
  column     = X >> 4
  rowFromTop = 0x0A - ((Y - 0x30) >> 4)
  index      = (rowFromTop << 4) | column
  address    = 0x6200 + index
  direction  = RAM[address] & 0x0F
```

Timing detail:

```text
source input     = frame[i-1]
reference result = frame[i]
```

For BFS/chase guidance, the previous/source frame position gives the clean prediction.

## Current limitations

- The source-path single-enemy replay still syncs release timing from MAME.
- `0x2EC7` random preferred[] base tuples are still trace-synced by design.
- Only safe/visible `0x477D` BFS/chase overrides are modeled in the replay.
- Player position and input are still trace-driven.
- Gates, timers, VRAM context, inactive slots, and multi-enemy frames are still synchronized from MAME.
- Multi-enemy transitions are not modeled yet; they are detected and resynchronized.
- Moving-player traces are not implemented yet.
- Pivoting-door interaction as a gameplay input is not yet part of the validation workflow.
- Chase timers and round-robin selection are still mostly trace-observed, though BFS guidance direction is now modeled for the validated mono-enemy scope.
- Exact-PC logs remain diagnostic aids only.

## Roadmap

Near-term priority:

```text
Validate one-enemy movement with a static player while reducing trace-synced inputs only where it helps the goal.
```

Current status for that priority:

```text
- source-path movement is modeled;
- deterministic rotate preferred[] is modeled;
- visible BFS/chase guidance overrides are modeled;
- random remains trace-synced intentionally, because exact Z80 R reproduction is not required for validating enemy movement logic.
```

Likely next steps:

```text
1. run v0.9.14c on a few more static-player traces if desired;
2. decide whether one-enemy/static-player movement is good enough to port/refactor into the main Godot game;
3. consider release timing / den-exit only if it blocks the next validation goal;
4. add traces where the player moves;
5. add explicit pivoting-door interaction traces;
6. add true multi-enemy source-path analysis later.
```
