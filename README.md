# Enemy Trace Simulator

Enemy Trace Simulator is a standalone **Godot .NET / C#** diagnostic tool used to validate enemy movement behavior for the arcade game **Lady Bug**.

The repository is separate from the main Lady Bug remake project. Its job is not to become a second implementation of Lady Bug. Its job is to act as a microscope:

```text
MAME trace = reference
Godot / C# simulation = candidate
Enemy Trace Simulator = visual and diagnostic comparison harness
```

For detailed implementation notes, source-address mappings, reverse-engineering status, and workflow details, see:

```text
doc/current_implementation.md
```

## Current status

Current checkpoint: **v0.9.12**

Latest stable milestone:

```text
Hybrid preferred[] provider for deterministic 0x2E97 rotate tuples
```

v0.9.12 keeps the source-path single-enemy replay as the default candidate and reduces one MAME trace dependency: preferred directions from the deterministic `0x2E97` rotate branch are now generated by the model and used by the replay.

The current validated scope is:

```text
validated scope    = one active enemy, static player, source-path movement replay
preferred[] scope  = 0x2E97 rotate modeled; random 0x2EC7 and BFS 0x477D still trace-synced
out-of-scope now   = autonomous release, autonomous random/BFS preferred[], moving player, true multi-enemy replay
future scope       = moving player, pivoting-door interaction, multi-enemy logic
```

The key progression is:

```text
Before v0.9.9:
  the simulation board advanced the enemy using the direction observed in the MAME trace.

Since v0.9.9b:
  the simulation board computes the one-enemy step with the reconstructed source path:
    0x427E decision gate
    -> 0x42E6 preferred / 0x4325 current / 0x4241 fallback
    -> 0x43BA one-pixel movement

Since v0.9.10b:
  this source-path replay is the default candidate exposed by the UI and adapter wrapper.

Since v0.9.12:
  preferred[] tuples produced by the deterministic 0x2E97 rotate branch are generated by the model and used by the replay.
```

Some inputs are still synchronized from MAME for this controlled milestone:

```text
release timing
random 0x2EC7 preferred[] cases
BFS/chase 0x477D preferred[] override cases
player position
gates
VRAM context
timers
inactive slots
multi-enemy frames after the scope limit
```

So this is not yet fully autonomous enemy AI, but the default candidate now models both the visible source-path movement step and a deterministic subset of preferred-direction generation.

## What the tool currently does

### MAME trace generation

The tool can launch MAME using:

```text
config/mame_trace_settings.json
```

The normal JSONL trace is generated by:

```text
tools/mame/lua/ladybug_sequence_trace.lua
```

Recommended trace settings for current one-enemy validation:

```json
{
  "luaScriptPath": "res://tools/mame/lua/ladybug_sequence_trace.lua",
  "outputPrefix": "ladybug_sequence_v8_fullmem",
  "framesAfterTick0": 600,
  "includeFullMemoryEachFrame": true,
  "includeLogicalMazeEachFrame": true,
  "enableDebugger": false
}
```

A longer trace is useful for detecting when the sequence leaves the one-enemy validation scope:

```json
{
  "luaScriptPath": "res://tools/mame/lua/ladybug_sequence_trace.lua",
  "outputPrefix": "ladybug_sequence_v8_fullmem_1200",
  "framesAfterTick0": 1200,
  "includeFullMemoryEachFrame": true,
  "includeLogicalMazeEachFrame": true,
  "enableDebugger": false
}
```

Do not enable the MAME debugger for the standard JSONL trace. Exact-PC diagnostics are a separate workflow.

### Visual replay

The main window displays two boards:

```text
left  = Simulation C# / Godot
right = Trace MAME
```

Playback controls:

```text
↺   restart from beginning
▶   resume playback
❚❚  pause playback
▶|  advance one tick
```

During playback, the simulator compares the currently displayed simulation state against the currently displayed MAME reference state.

If a mismatch is found:

```text
- playback pauses automatically;
- the display remains on the divergent tick;
- the status label reports the mismatch;
- the console prints the first mismatch details.
```

The visual comparator currently checks:

```text
player:  active, x, y, direction
enemies: active, x, y, direction
gates:   orientation and pivot
```

It intentionally ignores internal movement diagnostics such as:

```text
preferred[]
rejectedMask
fallback helper
chase timers
EnemyWork scratch values
```

Those fields are useful for analysis, but they should not stop the normal visual replay unless they affect visible behavior.

### Compare adapter

The current main Compare action is:

```text
Run Lady Bug source-path single-enemy replay
```

This mode is implemented through:

```text
LadyBugEnemySimulationAdapter
  -> LadyBugSourcePathSingleEnemyReplayAdapter
```

`LadyBugEnemySimulationAdapter` is a stable default wrapper. `LadyBugSourcePathSingleEnemyReplayAdapter` is the real implementation.

For one-active-enemy transitions, it computes the enemy movement step with the reconstructed source path:

```text
0x427E decision gate
  carry set:
    preferred 0x42E6
    current   0x4325 if preferred rejected
    fallback  0x4241 if current rejected
  carry clear:
    outside-center keep / forced reversal 0x433A
then:
  0x43BA one-pixel movement step
```

For preferred[] generation, v0.9.12 uses a hybrid provider:

```text
0x2E97 rotate branch  -> modeled when the model-generated tuple matches the trace tuple
0x2EC7 random branch  -> still trace-synced
0x477D BFS override   -> still trace-synced
```

It compares both visible state and modeled EnemyWork fields against the trace:

```text
modeledSlotMismatches
modeledEnemyWorkMismatches
modeledClean
fullScopeClean
firstModeledProblem
preferredModeledRotateTransitions
preferredTraceFallbackTransitions
```

When the trace leaves the one-enemy scope, the adapter resynchronizes from MAME and reports it:

```text
multiEnemySyncTransitions > 0
multiEnemyMode=reference-sync-after-scope-limit
```

## Important coordinate and direction conventions

### MAME-to-Godot vertical mirror

The simulator uses MAME RAM coordinates as the trace reference, but the Godot board displays them through the known vertical mirror:

```csharp
private const int MameYMirror = 0xDD;

private static int ToMameY(int godotArcadeY)
{
    return MameYMirror - godotArcadeY;
}
```

Equivalently:

```text
godotArcadeY = 0xDD - mameY
```

### Enemy direction encoding

Enemy movement uses this source encoding:

```text
01 = left
02 = up
04 = right
08 = down
```

Because of the MAME-to-Godot vertical mirror, source enemy directions must be mapped when comparing to `maze.json`:

```text
source 01/left  -> Godot maze left
source 02/up    -> Godot maze down
source 04/right -> Godot maze right
source 08/down  -> Godot maze up
```

This correction was validated in v0.9.6 and prevents false `sourceAcceptedButStaticBlocked` reports.

### Player direction / input encoding

Player direction/input uses a different vertical convention:

```text
01 = left
02 = down
04 = right
08 = up
```

Do not mix player and enemy direction encodings.

### Enemy decision center

Enemy decisions are aligned to:

```text
x & 0x0F == 0x08
y & 0x0F == 0x06
```

At each update the enemy moves by one arcade pixel. A new preferred/current/fallback decision is not taken at every pixel. Outside decision centers, the enemy normally keeps its direction, except for door/tile-related forced reversal cases.

## VRAM and maze conventions

### VRAM layout

Lady Bug background VRAM is column-major and bottom-to-top:

```text
D080-D09F = first column, bottom to top
D0A0-D0BF = second column, bottom to top
...
D3E0-D3FF = last column, bottom to top
```

Do not implement tile lookup as row-major screen order.

### `0x3C0A` tile lookup

Validated formula:

```text
HL = 0xD0A0 + ((D & 0xF8) * 4) + (E >> 3)
```

Meaning:

```text
D & F8  -> align probe X to 8-pixel tile column
* 4     -> convert to 32-byte column stride
E >> 3  -> tile index inside the column
D0A0    -> base used by this routine
```

### Logical maze and movement validator

The trace can include:

```text
6200..62AF = logical maze / navigation data
```

But the source-path movement validator uses:

```text
0x3911  logical maze validation through ROM table 0x0DA2
0x4130  local door / tile validation
```

Therefore, `0x6200..0x62AF` is useful diagnostic data, but it should not be treated as the complete final movement oracle.

## Historical notes

### v0.8 diagnostic bridge

v0.8.0 validated a replay bridge using exact-PC data such as `error.log`. This was useful but not final gameplay logic.

The v0.9 direction moved away from:

```text
MAME exact-PC logs -> runtime decisions -> perfect replay
```

toward:

```text
real Godot enemy logic -> candidate replay -> first divergence against MAME
```

### v0.9.6 correction

v0.9.6 corrected two important methodological issues:

```text
- inspect transitions frame[i-1] -> frame[i], not already committed frames;
- mirror vertical enemy directions when comparing source directions to maze.json.
```

### v0.9.8 correction

v0.9.8 made the source-path inspector multi-enemy safe by skipping unsupported multi-enemy transitions explicitly.

### v0.9.9b correction

v0.9.9 introduced the first source-path single-enemy replay adapter. v0.9.9b corrected the fallback rejected-mask behavior: directions rejected during the internal fallback scan are not added to `0x61C1`; only the preferred/current rejection path updates the rejected mask.

### v0.9.10b cleanup

v0.9.10 made the source-path single-enemy replay adapter the clarified default candidate. v0.9.10b renamed the Compare UI action so it no longer refers to the obsolete reference-direction label.

### v0.9.11 preferred[] preflight

v0.9.11 added a trace-sync preflight classifier for preferred[] tuples.

It verified that observed preferred[] tuples were explainable by these source families:

```text
0x2E97 rotate
0x2EC7 random / R-low
0x477D BFS/chase override
```

v0.9.11 did not remove trace-sync. It only validated coverage before the next step.

### v0.9.12 hybrid preferred provider

v0.9.12 started using modeled preferred[] data in the replay for the deterministic `0x2E97` rotate branch.

It still uses trace-sync for:

```text
0x2EC7 random / R-low cases
0x477D BFS/chase override cases
unmatched or unsupported preferred[] tuples
```

## Current limitations

- The source-path single-enemy replay still syncs release timing from MAME.
- preferred[] is only partially modeled: deterministic `0x2E97` rotate tuples are modeled, while random `0x2EC7` and BFS/chase `0x477D` cases are still trace-synced.
- Player position and input are still trace-driven.
- Gates, timers, VRAM context, inactive slots, and multi-enemy frames are still synchronized from MAME.
- Multi-enemy transitions are not modeled yet; they are detected and resynchronized.
- Moving-player traces are not implemented yet.
- Pivoting-door interaction as a gameplay input is not yet part of the validation workflow.
- Chase timers, round-robin behavior, full BFS/chase, and multi-enemy behavior are not independently validated yet.
- Exact-PC logs remain diagnostic aids only.

## Roadmap

Near-term priority:

```text
Validate one-enemy movement with a static player while reducing trace-synced inputs one by one.
```

This includes:

```text
- keep the clean two-board visual replay as the main workflow;
- validate source-path single-enemy replay on several static-player traces;
- reduce trace-synced inputs only when the source model is validated;
- stop at the first visible mismatch;
- use source-path and preferred[] diagnostics to explain any mismatch.
```

Likely next steps:

```text
1. investigate whether 0x2EC7 random/R-low preferred[] can become model-driven safely;
2. investigate BFS/chase 0x477D preferred[] overrides separately;
3. consider release timing / den-exit trace-sync after preferred[] is better understood;
4. add traces where the player moves;
5. add explicit pivoting-door interaction traces;
6. add true multi-enemy source-path analysis later.
```

Multi-enemy support is intentionally not the immediate priority. v0.9.8 and later only make the current code safe when a trace leaves the one-enemy scope.
